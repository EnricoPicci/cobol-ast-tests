# Role
You are a technical writer with expertise in open-source project documentation.

# Context
This is an educational project that teaches how to use AST (Abstract Syntax Tree) parsers to analyze and transform COBOL source code. The project has several distinctive characteristics:

1. **Prompt-driven development** — Every artifact in this project (code, documentation, even prompts themselves) was created by giving prompts to Claude Code. The `prompts/` directory contains numbered prompt files that trace the full history of how the project was built, step by step.
2. **Multi-language** — The project starts with Python and plans to add more languages. All languages share the same COBOL sample files.
3. **Educational-first** — Code, tests, and docs are written to teach, not just to function.

# Task
Generate (or regenerate) the `README.md` file at the project root. The README must serve a visitor who knows nothing about this project and help them understand what it is, why it exists, what they can learn from it, and how it was built. Include the following sections.

## Required Sections

### 1. Project Title and Introduction
- What the project is: an educational collection of examples for parsing COBOL code using AST parsers.
- Who it's for: developers curious about COBOL modernization, AST parsing, or AI-assisted development workflows.

### 2. Project Goal
- Explain that the goal is to learn and demonstrate how AST parsers can analyze (and potentially transform) COBOL source code.
- Mention the multi-language approach (Python first, with plans for TypeScript/Node.js, Go, Java).

### 3. Key Documentation
List and briefly describe the important documents a newcomer should read. Discover these by scanning the repository — look in `ast-docs/`, `cobol-docs/`, and any other documentation directories. For each document, include:
- The file path (as a relative link)
- A one-line description of what it covers

### 4. How This Project Is Built — Prompt-Driven Development
This is a distinguishing feature of the project and deserves its own section. Explain:
- Every artifact was generated via prompts given to Claude Code.
- Prompts are often themselves generated by prompts (meta-prompts).
- The `prompts/` directory contains numbered files (`01-...`, `02-...`, etc.) that record the exact sequence of instructions used to build the project.
- A reader can follow the numbered prompts to understand how the project evolved from scratch.
- Include a brief walkthrough: list each prompt file with a one-sentence description of what it does. Discover these by reading the prompt files.

### 5. Project Structure
Show the directory tree with short annotations for each top-level directory. Base this on the actual current structure of the repository — do not hardcode; inspect the file system.

### 6. Languages
For each language currently implemented:
- Its directory
- How to set up and run examples/tests (refer to the language-specific `CLAUDE.md` or its own README for details)

### 7. COBOL Samples
Explain that `samples/` contains shared COBOL source files used by all language implementations.

## Guidelines

- **Discover, don't assume** — Read the actual repository contents to populate each section. Do not invent or guess file names, document titles, or directory structures.
- **Keep it concise** — Each section should be short and scannable. Use bullet points and tables where appropriate.
- **Relative links** — Link to files and directories using relative paths so links work on GitHub.
- **No badges or boilerplate** — Skip CI badges, license badges, or generic open-source boilerplate unless they already exist in the project.
- **Tone** — Friendly, educational, approachable. This is a learning project, not a production system.

## Post-Generation Checklist

After writing the README, verify:
1. Every linked file actually exists in the repository.
2. The directory tree matches the real structure.
3. The prompt walkthrough includes all files currently in `prompts/`.
4. The language list matches the directories that actually exist.
